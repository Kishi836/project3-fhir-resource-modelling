## 1. Problem Statement

Healthcare providers in India increasingly need to exchange clinical records across systems — between CGHS-empanelled hospitals, private facilities, and the ABDM ecosystem. The vehicle for this exchange is FHIR R4: a structured, machine-readable format where every clinical concept maps to a specific "Resource" type with a defined JSON schema.

The problem today: most developers working on health applications either (a) don't know FHIR at all, or (b) know FHIR conceptually but have never authored, validated, or queried real FHIR resources. The gap between understanding FHIR and being able to *build* with FHIR costs Indian health startups weeks of onboarding time.

**The product we are building**: A complete, validated set of FHIR R4 resources modelling a real diabetic patient follow-up care pathway at a hypothetical AIIMS clinic — and a Python script that fetches these resources from a FHIR server and reconstructs a chronological clinical timeline.

## 6. Functional Requirements

### 🟢 MVP Features

#### F-01 · FHIR Resource Authoring (Manual JSON)
Students must author the following resources by hand (not generated by libraries):

- **F-01.1 Patient** (1 resource)
  - `resourceType: "Patient"`
  - `identifier`: ABHA ID with `system: "https://abdm.gov.in/ABHA"`
  - `name`, `birthDate`, `gender`, `address` (India-formatted)
  - `telecom`: phone number

- **F-01.2 Practitioner** (1 resource)
  - NMC registration number in `identifier`
  - Specialization in `qualification`

- **F-01.3 Encounter** (2 resources — OPD visit + admission)
  - `status`: finished
  - `class`: AMB (ambulatory) for OPD, IMP (inpatient) for admission
  - `subject.reference`: Patient reference
  - `participant.individual.reference`: Practitioner reference
  - `period.start` and `period.end`

- **F-01.4 Observation** (minimum 4 resources)
  - HbA1c: LOINC 4548-4
  - Fasting Blood Glucose: LOINC 1558-6
  - Systolic BP: LOINC 8480-6
  - Diastolic BP: LOINC 8462-4
  - Each must include: `code`, `valueQuantity` (value + unit), `subject.reference`, `encounter.reference`, `effectiveDateTime`, `status: "final"`

- **F-01.5 Condition** (2 resources)
  - T2DM: `code.coding.system: "http://hl7.org/fhir/sid/icd-10"`, code: `E11.9`
  - Hypertension: code: `I10`
  - `clinicalStatus`: active
  - `subject.reference`, `encounter.reference`

- **F-01.6 MedicationRequest** (2 resources)
  - Metformin 500mg BD
  - Amlodipine 5mg OD
  - `medicationCodeableConcept` with RxNorm or local code
  - `dosageInstruction`: dose, frequency, route
  - `subject.reference`, `encounter.reference`

- **F-01.7 AllergyIntolerance OR Procedure** (1 resource, student choice)
  - Adds breadth to the resource coverage

#### F-02 · FHIR Server Validation
- **F-02.1** POST each resource to `https://hapi.fhir.org/baseR4/{ResourceType}` in the correct order (Patient first, then Practitioner, then Encounters, then linked resources)
- **F-02.2** Capture the HTTP status code and Location header for each POST
- **F-02.3** For any HTTP 422 response: parse the `OperationOutcome` and document the issue
- **F-02.4** Replace placeholder `id` references with server-assigned IDs after creation
- **F-02.5** Generate a `validation_report.txt` with: resource name, HTTP status, server-assigned ID, any issues

#### F-03 · Clinical Timeline Builder (Python)
- **F-03.1** Accept a patient ID (ABHA ID or HAPI-assigned ID) as input
- **F-03.2** Fetch all resources for the patient using FHIR search:
  - `GET /Patient?identifier=<abha_id>`
  - `GET /Encounter?patient=<id>`
  - `GET /Observation?patient=<id>&_sort=date`
  - `GET /Condition?patient=<id>`
  - `GET /MedicationRequest?patient=<id>`
- **F-03.3** Merge all resources into a chronological list sorted by `effectiveDateTime` or `period.start`
- **F-03.4** Print a formatted timeline to console:
  ```
  ── 2024-01-15  OPD Visit (Dr. Gupta, AIIMS Endocrinology)
     Conditions: T2DM (E11.9), Hypertension (I10)
     Labs: HbA1c 9.2%, FBG 185 mg/dL
     Medications started: Metformin 500mg BD, Amlodipine 5mg OD
  ── 2024-02-26  OPD Follow-up
     Vitals: BP 128/82 mmHg, FBG 142 mg/dL
  ── 2024-03-10  Inpatient Admission (3 days)
     Reason: Acute hyperglycaemia (BG 380 mg/dL)
     Medications changed: Insulin added
  ── 2024-03-13  Discharge
  ```
- **F-03.5** Export timeline as a JSON file `patient_timeline.json` with structured event objects

## 10. 🤖 GenAI / LLM Requirements

> These requirements explore how AI can accelerate FHIR development and enhance clinical intelligence.

### GA-01 · FHIR Resource Generator from Clinical Notes

**Problem**: Authoring FHIR JSON manually is error-prone and time-consuming. In production, clinical notes arrive as free text and must be converted to structured FHIR resources.

**LLM Solution**: Given a free-text clinical note, extract structured data and generate a valid FHIR R4 Observation bundle.

```python
def clinical_note_to_fhir(note: str, patient_id: str) -> list[dict]:
    prompt = f"""
    Extract all lab results and vital signs from this clinical note and format them as FHIR R4 Observation resources.
    
    Clinical note: "{note}"
    Patient ID: "{patient_id}"
    
    For each measurement:
    - Use correct LOINC code (look up from standard LOINC table)
    - Include value, unit, and effectiveDateTime
    - Set status: "final"
    - Format as valid FHIR R4 JSON
    
    Return a JSON array of Observation resources.
    """
    # Returns list of FHIR Observation JSON objects

# Example input: "HbA1c 9.2%, FBG 185 mg/dL, BP 145/90 mmHg on 15-Jan-2024"
# Example output: [{"resourceType":"Observation","code":{"coding":[{"system":"http://loinc.org","code":"4548-4",...}]}}]
```

**Acceptance Criteria**: LLM-generated resources POST to HAPI with HTTP 201 for 80% of test inputs.

### GA-02 · Clinical Timeline Narrative Generator

**Problem**: The timeline builder outputs structured JSON that only developers can read. Clinicians need a readable clinical summary.

**LLM Solution**: Feed the `patient_timeline.json` to an LLM and receive a clinical narrative in the format of a referral letter or handover note.

```python
def timeline_to_clinical_narrative(timeline: dict, format_type: str = "referral_letter") -> str:
    prompt = f"""
    You are writing a clinical handover note for an Indian endocrinologist.
    
    Patient timeline data: {json.dumps(timeline)}
    Format: {format_type}  (one of: referral_letter / handover_note / patient_summary)
    
    Write in clinical English. Include:
    - Disease history with timeline
    - Key investigations and their trends
    - Current medications
    - Clinical impression
    - Pending actions
    
    Do not fabricate clinical information not present in the timeline data.
    """
```

**Use case**: ABDM's Health Information Exchange sends FHIR Bundle → PHR app receives it → displays a human-readable summary to the patient.

### GA-03 · FHIR Resource Validator & Explainer

**Problem**: When HAPI returns a 422 error with an `OperationOutcome`, students (and developers) often can't understand what the error means or how to fix it.

**LLM Solution**: Parse the `OperationOutcome` and generate a plain-English explanation with a specific fix.

```python
def explain_fhir_validation_error(operation_outcome: dict, resource_json: dict) -> dict:
    prompt = f"""
    A FHIR server returned this validation error for a resource I submitted.
    
    OperationOutcome: {json.dumps(operation_outcome)}
    Resource submitted: {json.dumps(resource_json)}
    
    Explain:
    1. What is wrong (in plain English, not FHIR jargon)
    2. Which field needs to be changed
    3. Provide the corrected JSON snippet for that field only
    """
    # Returns: {explanation, field, corrected_snippet}
```

**Learning value**: Turns cryptic FHIR errors into teachable moments. Particularly useful for common errors like missing `status` field, wrong `system` URL, or invalid code values.

### GA-04 · LOINC Code Recommendation Engine

**Problem**: Choosing the right LOINC code from 90,000+ options is a major friction point in FHIR development. Developers frequently use incorrect or overly general codes.

**LLM Solution**: Given a lab test description, recommend the most specific LOINC code.

```python
def recommend_loinc_code(test_description: str) -> dict:
    prompt = f"""
    Recommend the most specific LOINC code for this lab test.
    Test: "{test_description}"
    
    Return JSON: {{
        "loinc_code": "4548-4",
        "display_name": "Hemoglobin A1c/Hemoglobin.total in Blood",
        "specimen": "Blood",
        "method": "HPLC",
        "confidence": "high",
        "alternative_codes": ["17856-6"],
        "why_this_code": "..."
    }}
    """

# Test: "blood HbA1c" → LOINC 4548-4
# Test: "urine albumin to creatinine ratio" → LOINC 9318-7
# Test: "random blood sugar" → LOINC 2345-7
```

### GA-05 · FHIR Diff & Change Detection

**Problem**: When a patient's records are updated (e.g., medication changed, new diagnosis added), clinicians need to quickly understand what changed between two FHIR Bundle versions.

**LLM Solution**: Compare two `patient_timeline.json` snapshots and generate a clinical change summary.

```python
def clinical_diff(timeline_v1: dict, timeline_v2: dict) -> str:
    prompt = f"""
    Compare these two patient clinical timelines and describe what changed.
    
    Previous state: {json.dumps(timeline_v1)}
    Current state: {json.dumps(timeline_v2)}
    
    Summarize changes in plain clinical language:
    - New diagnoses added
    - Medications changed or discontinued
    - Lab values trending up or down
    - New procedures or encounters
    """
```

## 12. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| HAPI sandbox downtime during submission | Medium | High | Submit 48 hrs before deadline; screenshot HTTP 201 responses |
| Students use wrong LOINC system URL | High | Medium | Provide correct URL: `"http://loinc.org"` in starter template |
| References break after HAPI assigns new IDs | High | High | Script must update references after each POST |
| LLM generates syntactically valid but clinically wrong FHIR | Medium | Medium | Always validate against HAPI before accepting LLM output |
| ABHA ID format not matching ABDM specification | Medium | Low | Provide ABDM identifier system URL in starter resources |
